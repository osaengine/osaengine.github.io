Мы опубликовали на Хабре подробную статью ["Брокерские торговые API в России в 2025 году"](https://habr.com/ru/articles/963856/), где проанализировали технологические решения для автоматизированной торговли у четырёх крупнейших российских брокеров.

## Что внутри?

В статье мы разобрали текущее состояние торговых API для создания торговых роботов, разделив решения по уровню надёжности и готовности к продакшену.

### Финам: два мира в одном

Двухуровневая архитектура с проверенным временем **Transaq/TXmlConnector** как надёжным решением для боевых систем и новым **Finam Trade API** (gRPC/REST/WebSocket) как развивающейся современной альтернативой.

### Алор: лидер открытости

Ранняя ставка на открытый **WebAPI** превратила решение в зрелый продукт. **ALOR OpenAPI** стал де-факто стандартным способом подключения роботов у клиентов брокера.

### Т-Банк: амбиции vs стабильность

**T-Invest API** — амбициозный проект с современной архитектурой, но критическая проблема нестабильности соединения делает его пока экспериментальным решением. Для серьёзной торговли не подходит.

### БКС: переходный период

**Trade API** находится на этапе трансформации из внутреннего инструмента в публичный продукт. Требует осторожного использования и тестирования.

## Главные выводы

Рынок API развивается консервативно: старые решения не вытесняются мгновенно, новые технологии дозревают медленно.

**Для боевого продакшена рекомендуются:**
- Transaq (Финам)
- ALOR OpenAPI (Алор)

**Для экспериментов подходят:**
- Finam Trade API
- T-Invest API
- БКС Trade API

## Ключевая мысль

При выборе API для серьёзных торговых систем **стабильность важнее современности архитектуры**. Красивые технологии не помогут, если робот отключится в момент открытой позиции.

Статья будет полезна всем, кто планирует создавать торговых роботов на российском рынке или уже работает с брокерскими API.
